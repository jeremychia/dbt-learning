dbt Snapshots

Step through Type-2 SCD mechanics, compare timestamp vs. check strategies, and see exactly what SQL dbt generates on every run.

Concept
The problem snapshots solve
Source tables overwrite rows in place. Snapshots capture history before it's lost.

Source table — mutable

The order was updated. Yesterday's state is gone.

NULL = current record. Non-null dbt_valid_to means that row is historical.

Type-2 SCD mechanics: three snapshot events

New row

ID not in snapshot yet. dbt inserts it with dbt_valid_from = NOW() and dbt_valid_to = NULL.

Changed row

ID found but value differs. dbt closes the old record (dbt_valid_to = NOW()) and inserts a new current record.

Unchanged row

ID found, values identical. dbt leaves the record untouched. dbt_valid_to stays NULL.

Metadata

    
Columns dbt adds automatically

    
Every snapshot row gets these four (or five) fields generated by the snapshot macro.

  


  

    

      
dbt_scd_id

      
A deterministic MD5 hash of the concatenated unique_key and dbt_updated_at. Globally unique per snapshot row — useful as a surrogate key.

      
MD5(
  CAST(unique_key AS TEXT)
  || '||' ||
  CAST(dbt_updated_at AS TEXT)
)

    

    

      
dbt_valid_from

      
When this version of the row became active. On first insert: the source's updated_at (timestamp strategy) or NOW() (check strategy).

    

    

      
dbt_valid_to

      
NULL means the row is currently active. Set to the next run's timestamp when the row changes. Set a future sentinel instead of NULL with dbt_valid_to_current.

      
Tip: Use WHERE dbt_valid_to IS NULL to get only the latest version of each row.

    

    

      
dbt_updated_at

      
The timestamp from the source record. For timestamp strategy: copied from your updated_at column. For check strategy: set to NOW() at run time.

    

    

      
dbt_is_deleted

      
Only present when hard_deletes: 'new_record'. True for rows whose source ID disappeared. Lets you distinguish a "row was deleted" state from historical versions.

    

  


  

    
Customise meta-column names (dbt Core 1.9+)

    
If your downstream models already use these column names for something else, rename them:


snapshots:
  - name: orders_snapshot
    config:
      snapshot_meta_column_names:
        dbt_valid_from: start_date
        dbt_valid_to:   end_date
        dbt_scd_id:     scd_id
        dbt_updated_at: modified_date
        dbt_is_deleted: is_deleted

  







  

    
Strategies

    
timestamp vs. check — when to use each

    
Both strategies detect row changes, but they use fundamentally different mechanisms. The macro dispatch is snapshot_timestamp_strategy / snapshot_check_strategy.

  


  


    
    

      
timestamp strategy

      
Macro: snapshot_timestamp_strategy (strategies.sql)

      
Compares the source updated_at against the snapshot's dbt_valid_from. If source.updated_at != snapshot.dbt_valid_from, the row is considered changed.


snapshots:
  - name: orders_snapshot
    relation: ref('stg_orders')
    config:
      schema:     snapshots
      unique_key: order_id
      strategy:   timestamp
      updated_at: updated_at  # required

      

        
Row-changed expression (compiled)


source.updated_at != snapshot.dbt_valid_from
-- any difference (not just newer) is treated as a change

      

      
Recommended. Cheap to compute — compares one column. Robust to schema changes because it doesn't enumerate watched columns.

      
Requires: a reliable updated_at column that is always updated when any field changes. If the source sets updated_at inconsistently, silent misses occur.

    


    
    

      
check strategy

      
Macro: snapshot_check_strategy (strategies.sql)

      
Compares specific column values between source and snapshot. A row is changed if any watched column differs (including NULL transitions).


snapshots:
  - name: orders_snapshot
    relation: ref('stg_orders')
    config:
      schema:     snapshots
      unique_key: order_id
      strategy:   check
      check_cols: # watch specific columns
        - status
        - amount
      # or: check_cols: all

      

        
Row-changed expression (compiled for status + amount)


-- generated by snapshot_check_strategy macro
NOT (
  (source.status = snapshot.status
   OR (source.status IS NULL AND snapshot.status IS NULL))
  AND
  (source.amount = snapshot.amount
   OR (source.amount IS NULL AND snapshot.amount IS NULL))
)
-- NULL-safe: NULL → 'paid' is detected as a change

      

      
Use when: source table has no reliable updated_at. Lets you watch any subset of columns.

      
Pitfall: check_cols: all is brittle — adding a column to the source automatically starts tracking it, which can cause spurious snapshot rows if a new column always differs from the snapshot schema.

Compiled SQL

    
What dbt actually runs in the warehouse

    
Four source files in dbt-adapters: snapshot.sql orchestrates the run; strategies.sql detects changes; helpers.sql builds the staging table; snapshot_merge.sql applies it.

  


  
  

    
First run — CREATE AS SELECT

    
Macro: build_snapshot_table (helpers.sql) → plain CREATE TABLE

    

      

        
snapshot definition


-- snapshots/orders_snapshot.yml
snapshots:
  - name: orders_snapshot
    relation: ref('stg_orders')
    config:
      schema:     snapshots
      unique_key: order_id
      strategy:   timestamp
      updated_at: updated_at

      

      

        
compiled SQL (first run)


-- build_snapshot_table: initial load
CREATE TABLE snapshots.orders_snapshot AS
SELECT
  source.*,
  MD5(CAST(order_id AS TEXT)
      || '||'
      || CAST(updated_at AS TEXT))
      AS dbt_scd_id,
  updated_at    AS dbt_updated_at,
  updated_at    AS dbt_valid_from,
  NULL          AS dbt_valid_to
FROM stg_orders AS source;

      

    

  


  
  

    
Subsequent run — staging + merge

    
Macros: build_snapshot_staging_table (helpers.sql) → snapshot_merge_sql (snapshot_merge.sql)

    

      

        
step 1 — build staging CTE


-- build_snapshot_staging_table (helpers.sql)
-- detects inserts, updates, hard deletes
WITH snapshot_query AS (
  -- your SELECT from stg_orders
  SELECT * FROM stg_orders
),
snapshotted_data AS (
  -- current rows in snapshot table
  SELECT * FROM snapshots.orders_snapshot
  WHERE dbt_valid_to IS NULL
),
insertions_source_data AS (
  -- rows changed or new
  SELECT
    source.*,
    MD5(...) AS dbt_scd_id,
    source.updated_at AS dbt_updated_at,
    source.updated_at AS dbt_valid_from,
    NULL              AS dbt_valid_to
  FROM snapshot_query source
  LEFT JOIN snapshotted_data snap
    ON source.order_id = snap.order_id
  WHERE
    snap.order_id IS NULL  -- new
    OR source.updated_at != snap.dbt_valid_from
    -- ^^^ row_changed expression (timestamp strategy)
)

      

      

        
step 2 — merge into snapshot table


-- snapshot_merge_sql (snapshot_merge.sql)
MERGE INTO snapshots.orders_snapshot AS DBT_INTERNAL_DEST
USING staging_table AS DBT_INTERNAL_SOURCE
  ON DBT_INTERNAL_SOURCE.dbt_scd_id
     = DBT_INTERNAL_DEST.dbt_scd_id

-- close expired records
WHEN MATCHED
  AND DBT_INTERNAL_SOURCE.dbt_change_type = 'update'
  THEN UPDATE SET
    dbt_valid_to = DBT_INTERNAL_SOURCE.dbt_valid_to

-- insert new / changed records
WHEN NOT MATCHED
  AND DBT_INTERNAL_SOURCE.dbt_change_type = 'insert'
  THEN INSERT (order_id, status, updated_at,
               dbt_scd_id, dbt_updated_at,
               dbt_valid_from, dbt_valid_to)
       VALUES (DBT_INTERNAL_SOURCE.order_id, ...);

      

    

    
Key insight: The merge key is dbt_scd_id, not unique_key. This means the same order_id can have multiple rows in the snapshot table (one per version) — they each have a different dbt_scd_id.

  


  
  

    
Hard deletes

    
Config: hard_deletes: 'new_record' — adds a dbt_is_deleted column

    

      

        
By default, if a row disappears from the source, the snapshot doesn't know — the last version just stays open (NULL dbt_valid_to) forever.

        
With hard_deletes: 'new_record', dbt inserts a deletion record:


snapshots:
  - name: orders_snapshot
    config:
      strategy: timestamp
      updated_at: updated_at
      hard_deletes: new_record
      # also available: 'invalidate'
      # (closes the row, no deletion record)

Pitfalls

    
What can go wrong — and how to fix it

    
The most common snapshot mistakes and how to detect them early.

  


  
  

    

      
Silent misses with timestamp strategy

      Affects: timestamp strategy
    

    

      

        
What breaks

        
Source updates a row but doesn't change updated_at (bug in upstream pipeline, manual edit). The snapshot never detects the change — the stale value stays silently forever.

      

      

        
Detect

        
Periodically compare snapshot current rows against source on columns that matter. Add a dbt test that asserts the snapshot matches source for all non-updated_at columns.

      

      

        
Fix

        
Validate upstream updated_at is always updated on writes. If the source is unreliable, switch to check strategy on the important columns, or add updated_at in a staging model using CURRENT_TIMESTAMP.

      

    

  


  
  

    

      
Duplicate unique_key in source

      Affects: all strategies
    

    

      

        
What breaks

        
If unique_key is not unique in the source, the staging CTE produces multiple "current" rows for the same ID. The merge then produces duplicate snapshot rows — a broken SCD2 that can't be queried correctly.

      

      

        
Detect


-- schema.yml on the source
sources:
  - name: app_db
    tables:
      - name: orders
        columns:
          - name: order_id
            tests:
              - unique
              - not_null

      

      

        
Fix

        
Deduplicate in a staging model before the snapshot reads it. Use QUALIFY ROW_NUMBER() OVER (PARTITION BY order_id ORDER BY updated_at DESC) = 1.

      

    

  


  
  

    

      
Schema change breaks snapshot

      Affects: all strategies
    

    

      

        
What breaks

        
Adding a new column to the source: dbt automatically adds it to the snapshot table on the next run. Removing a column from the source: dbt will NOT remove it from the snapshot — old rows keep the stale column. Column type changes (INT → BIGINT) can silently cast or error.

      

      

        
Detect

        
Run dbt compile --select orders_snapshot and diff the compiled SQL before deploying. Monitor information_schema.columns for unexpected changes in the snapshot schema.

      

      

        
Fix

        
For breaking schema changes, drop and recreate the snapshot table (DROP TABLE snapshots.orders_snapshot then re-run). Accept loss of history, or migrate old rows manually.

      

    

  


  
  

    

      
Querying snapshots incorrectly

      Affects: downstream models
    

    

      

        
Wrong — returns duplicates


-- ❌ returns ALL versions of every order
SELECT * FROM orders_snapshot;

        
Order 1 shows up twice — once as "pending", once as "shipped".

      

      

        
Correct query patterns


-- ✅ current state only
SELECT * FROM orders_snapshot
WHERE dbt_valid_to IS NULL;

-- ✅ state at a point in time
SELECT * FROM orders_snapshot
WHERE dbt_valid_from <= '2024-01-02'
  AND (dbt_valid_to > '2024-01-02'
       OR dbt_valid_to IS NULL);

-- ✅ easier with dbt_valid_to_current='9999-12-31'
SELECT * FROM orders_snapshot
WHERE '2024-01-02' BETWEEN
      dbt_valid_from AND dbt_valid_to;

Point-in-time queries

    
Reconstructing state at any moment in history

    
The core use case for snapshots — and where most query mistakes happen.

  


  
  

    
The pattern

    
To get the state of a row as it was on a given date, filter for the version whose validity window contains that date:

    

      

        
with NULL (default)


SELECT *
FROM orders_snapshot
WHERE dbt_valid_from <= '2024-01-02'
  AND (
    dbt_valid_to  > '2024-01-02'
    OR dbt_valid_to IS NULL
  );
-- returns one row per unique_key

      

      

        
with dbt_valid_to_current = '9999-12-31'


SELECT *
FROM orders_snapshot
WHERE '2024-01-02'
      BETWEEN dbt_valid_from
          AND dbt_valid_to;
-- cleaner — no NULL handling needed
-- requires dbt_valid_to_current config

      

    

    
The most common mistake: forgetting OR dbt_valid_to IS NULL. This silently drops all currently-active rows from results — you only get historical versions.

Joining two snapshot tables at a point in time

    
Each snapshot needs its own independent point-in-time filter. Only filtering one side of the join is a very common mistake — it fans out to every historical version on the unfiltered side.

    

      

        
Wrong — only one side filtered


SELECT o.order_id, o.status,
       c.name AS customer_name
FROM orders_snapshot o
JOIN customers_snapshot c
  ON o.customer_id = c.customer_id
WHERE o.dbt_valid_to IS NULL;
-- ❌ joins ALL versions of customer
--    against each current order

      

      

        
Correct — both sides filtered


SELECT o.order_id, o.status,
       c.name AS customer_name
FROM orders_snapshot o
JOIN customers_snapshot c
  ON  o.customer_id = c.customer_id
  -- filter customer at same point in time
  AND '2024-01-04' BETWEEN
      c.dbt_valid_from AND
      COALESCE(c.dbt_valid_to, '9999-12-31')
-- filter orders at same point in time
WHERE '2024-01-04' BETWEEN
      o.dbt_valid_from AND
      COALESCE(o.dbt_valid_to, '9999-12-31');

      

    

  


  
  

    
Encapsulating the pattern in a dbt model

    
Rather than writing the filter everywhere, expose a current_orders model that downstream consumers can use safely:

    

      

        
models/current_orders.sql


-- always returns current state
SELECT *
FROM {{ ref('orders_snapshot') }}
WHERE dbt_valid_to IS NULL

      

      

        
models/orders_at.sql (parameterised)


-- pass a var for the report date
SELECT *
FROM {{ ref('orders_snapshot') }}
WHERE dbt_valid_from <= '{{ var("report_date") }}'
  AND (
    dbt_valid_to  > '{{ var("report_date") }}'
    OR dbt_valid_to IS NULL
  )

      

    

    
Tip: Build current_orders as a regular view materialization — it always reflects the latest snapshot state without any storage cost.

  







  

    
BI tools

    
Exposing snapshot history in Looker (and other tools)

    
Snapshot tables are multi-row per entity — BI tools expect one row per entity. The translation layer lives in dbt, not in Looker.

  


  
  

    
Why you can't just point Looker at the snapshot table

    

      

        
A snapshot table has multiple rows per order. If a Looker explore joins orders to line items on order_id, every historical version fans out — metrics like revenue are multiplied by version count.

        
Fan-out: order 1 has 2 versions × 3 line items = 6 rows instead of 3. Revenue doubles.

Three patterns — choose based on what Looker needs

    


      

        
Pattern 1 — Current state view

        
A dbt view that filters to dbt_valid_to IS NULL. Looker treats it like a normal table — one row per order. Use this for 90% of dashboards.


-- models/current_orders.sql
SELECT * EXCEPT (
  dbt_scd_id,
  dbt_valid_from,
  dbt_valid_to,
  dbt_updated_at
)
FROM {{ ref('orders_snapshot') }}
WHERE dbt_valid_to IS NULL

        
Best for: operational dashboards, current KPIs

      


      

        
Pattern 2 — Date spine join

        
Join the snapshot to a date spine so each row represents one entity on one day. Looker can then filter by date and show how metrics changed over time.


-- models/orders_daily.sql
SELECT
  d.date_day,
  o.order_id,
  o.status,
  o.amount
FROM {{ ref('dim_dates') }} d
JOIN {{ ref('orders_snapshot') }} o
  ON d.date_day BETWEEN
     o.dbt_valid_from AND
     COALESCE(o.dbt_valid_to,
              CURRENT_DATE)

        
Best for: trend charts, "status over time" analyses

      


      

        
Pattern 3 — Looker liquid date

        
Let the Looker user pick the "as-of date" via a filter. Pass it into the LookML sql_table_name using a {% raw %}{% if %}{% endraw %} block or a derived table with a liquid variable.


-- LookML derived table
derived_table: {
  sql:
    SELECT * FROM orders_snapshot
    WHERE dbt_valid_from
          <= {% raw %}{{
      _filters['orders.report_date']
    }}{% endraw %}
    AND (dbt_valid_to
         > {% raw %}{{
      _filters['orders.report_date']
    }}{% endraw %}
    OR dbt_valid_to IS NULL) ;;
}

        
Best for: self-serve "what did this look like on date X?" explorers

      

    

  


  
  

    
Looker-specific gotchas

    

      

        
Symmetric aggregates

        
If you expose the raw snapshot in Looker and join line items, symmetric aggregates won't save you from fan-out — the duplication happens in the join before aggregation. Filter first in a derived table or dbt model.

      

      

        
PDT caching vs. live snapshots

        
Looker PDTs cache results. If your snapshot runs hourly but the PDT is cached for 24h, the Looker view is stale. Either trigger PDT rebuilds after dbt snapshot completes, or use a view-materialized dbt model (always fresh).

      

      

        
dbt_valid_to in Looker dimensions

        
Expose dbt_valid_from and dbt_valid_to as dimensions in LookML so users can see how long a status lasted. Hide dbt_scd_id and dbt_updated_at — they're internals.

      

      

        
Other BI tools

        
Same principles apply to Tableau, Power BI, Metabase. The safest pattern is always: filter in dbt, expose a clean view. Avoid relying on BI-layer filtering for deduplication — it's easy to forget and hard to test.
