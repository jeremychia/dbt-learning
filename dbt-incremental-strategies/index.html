Risk Mitigation

    
What can go wrong — and how to catch it

    
Risks grouped by category. Each entry: what breaks, how to detect it before it hits production, and the fix.

  


  
  

    

      🔁
      
Duplicate rows

      Affects: append · merge (no unique_key) · delete+insert (predicate trap)
    

    

      

        
What breaks

        
    
          
  • • append without filter — every run re-inserts the full source. N runs = N copies of every row.
    • 
          
  • • merge without unique_key — no MATCHED clause generated; behaves like append.
    • 
          
  • • delete+insert + predicates + no filter — old rows survive the DELETE (predicate blocks it) then are re-inserted.
    • 
          
  • • source duplicates on unique_key — BigQuery silently picks one; Snowflake errors; Spark behaviour is undefined.
    • 
        

      

      

        
Detect

        
    
          
  • • Add a dbt test: unique on your unique_key column after every run.
    • 
          
  • • Monitor row count delta — a sudden 2× jump is a duplicate signal.
    • 
          
  • • dbt-expectations: expect_column_values_to_be_unique with alerting on failure.
    • 
        


-- schema.yml
models:
  - name: orders
    columns:
      - name: order_id
        tests:
          - unique
          - not_null

      

      

        
Fix

        
    
          
  • • Always use is_incremental() filter with append and delete+insert.
    • 
          
  • • Deduplicate source before the strategy runs:
    • 
        


select *
from source
qualify
    row_number() over (
        partition by order_id
        order by updated_at desc
    ) = 1

        
    
          
  • • If duplicates already exist: dbt run --full-refresh rebuilds clean.
    • 
        

      

    

  


  
  

    

      🕰️
      
Stale data (silent updates missed)

      Affects: merge · delete+insert · insert_overwrite — all when predicate window is too narrow
    

    

      

        
What breaks

        
    
          
  • • A row changes in source but its date falls outside the predicate window — the merge ON clause never sees it. Target retains old value forever, no error raised.
    • 
          
  • • Late-arriving events (e.g. mobile offline events syncing 48h later) miss a filter that only looks back 24h.
    • 
          
  • • insert_overwrite with a short lookback silently drops restated rows on old partitions.
    • 
        

      

      

        
Detect

        
    
          
  • • Periodically compare target against source for rows within the last N days:
    • 
        


-- Staleness check: rows in source
-- that differ from target
select s.order_id
from source s
join target t using (order_id)
where s.updated_at > t.updated_at
  and s.updated_at
      > current_date - interval '30 days'

        
    
          
  • • dbt source_freshness checks flag when source hasn't updated recently.
    • 
        

      

      

        
Fix

        
    
          
  • • Size predicate window to your actual late-arrival SLA, not a round number. If data can arrive 3 days late, use 4+ days.
    • 
          
  • • Filter on updated_at, not event_date — an event from last month can have a recent updated_at if it was corrected.
    • 
          
  • • For high-correctness requirements: run a weekly full-refresh as a safety net alongside daily incrementals.
    • 
          
  • • microbatch lookback config is the cleanest solution for time-series late arrivals.
    • 
        

      

    

  


  
  

    

      📐
      
Schema drift (silent column loss)

      Affects: all strategies — controlled by on_schema_change, not the strategy itself
    

    

      

        
What breaks

        
    
          
  • • Default on_schema_change='ignore' silently excludes new columns from the INSERT. The column exists in the model but not in the target — all values are NULL.
    • 
          
  • • Removed columns stay in the target with stale data unless sync_all_columns is used.
    • 
          
  • • Column type changes (e.g. INT → BIGINT) may silently cast or error depending on the warehouse.
    • 
        

      

      

        
Detect

        
    
          
  • • dbt test: not_null on columns that should always be populated — will catch the silent-NULL case.
    • 
          
  • • Compare information_schema.columns on source vs target after each run in a post-hook.
    • 
          
  • • Set on_schema_change='fail' in development — forces you to think about schema changes before they reach prod.
    • 
        

      

      

        
Fix

        
    
          
  • • Choose the right on_schema_change for your team's risk tolerance:
    • 
        


{{ config(
  materialized='incremental',
  on_schema_change='append_new_columns'
  -- adds new cols; old cols stay
) }}

        
    
          
  • • For breaking changes: run --full-refresh to rebuild the table with the new schema.
    • 
          
  • • Document column changes in a migration log so downstream consumers know what changed and when.
    • 
        

      

    

  


  
  

    

      ⚡
      
Partial failure (data loss mid-run)

      Affects: delete+insert (non-atomic) · microbatch (batch-level failures)
    

    

      

        
What breaks

        
    
          
  • • delete+insert: DELETE succeeds, INSERT fails (OOM, timeout, network). Matched rows are permanently deleted with no replacement. Table is in an incomplete state until re-run.
    • 
          
  • • microbatch: a batch fails after deleting the target window but before re-inserting. That time window is empty in the target.
    • 
          
  • • Leftover __dbt_tmp tables block the next run with "relation already exists".
    • 
        

      

      

        
Detect

        
    
          
  • • Monitor row count after each run — a sudden drop (not just slower growth) signals a partial delete.
    • 
          
  • • Alert on dbt run failures immediately; don't let a partial state persist until the next scheduled run.
    • 
          
  • • Query information_schema.tables for orphaned __dbt_tmp tables as a health check.
    • 
        


-- Find orphaned temp tables
select table_name
from information_schema.tables
where table_name like '%__dbt_tmp'
  and table_schema = 'your_schema'

      

      

        
Fix

        
    
          
  • • Prefer merge over delete+insert when the platform supports it — MERGE is a single atomic statement; failure leaves the target unchanged.
    • 
          
  • • Wrap delete+insert in an explicit transaction if your warehouse supports it (Postgres, Redshift):
    • 
        


-- post-hook or pre-hook approach
begin;
  delete from {{ this }} where ...;
  insert into {{ this }} select ...;
commit;

        
    
          
  • • On failure: drop orphaned temp tables, then re-run. microbatch retries only the failed batch automatically on the next run.
    • 
        

      

    

  


  
  

    

      💸
      
Unexpected cost overrun

      Affects: all strategies — particularly merge without predicates, microbatch with long history, full-refresh
    

    

      

        
What breaks

        
    
          
  • • Merge without incremental_predicates full-scans the target on every run — expensive at 100M+ rows.
    • 
          
  • • --full-refresh on a microbatch model with begin='2020-01-01' generates 1,400+ individual SQL round-trips.
    • 
          
  • • insert_overwrite lookback set too wide rewrites far more partitions than necessary each run.
    • 
          
  • • Removing is_incremental() filter accidentally causes full source scan on every run.
    • 
        

      

      

        
Detect

        
    
          
  • • Track bytes scanned and execution time per model in your warehouse's query history — a sudden spike on an incremental model signals a filter regression.
    • 
          
  • • Set warehouse cost alerts on query spend thresholds (Snowflake resource monitors, BigQuery budget alerts).
    • 
          
  • • Run dbt compile and inspect the compiled SQL before running expensive models — confirm the WHERE clause is present.
    • 
        

      

      

        
Fix

        
    
          
  • • Always pair large-table merge with incremental_predicates.
    • 
          
  • • Set begin on microbatch models as late as your use case permits.
    • 
          
  • • Treat --full-refresh as a deliberate operation, not a default fix — require team sign-off for prod models over a certain size.
    • 
          
  • • Use CI/CD to run dbt compile --select model_name and diff the compiled SQL against the previous version — catch accidental filter removal before it reaches prod.
    •

Compiled SQL

    
What each strategy actually runs in the warehouse

    
Macro → compiled SQL → edge cases. All examples use our orders dataset.

  


  
  

    
append

    
Macro: get_incremental_append_sql → plain INSERT

    

      

        
dbt model


{{ config(
  materialized='incremental',
  incremental_strategy='append'
) }}

SELECT id, date, name, status, amt
FROM orders_source
{% if is_incremental() %}
  WHERE updated_at > (SELECT MAX(updated_at) FROM {{ this }})
{% endif %}

      

      

        
compiled SQL (incremental run)


-- Step 1: build temp table
CREATE TABLE orders__dbt_tmp AS
SELECT id, date, name, status, amt
FROM orders_source
WHERE date > (SELECT MAX(date) FROM orders);

-- Step 2: append (get_incremental_append_sql)
INSERT INTO orders (id, date, name, status, amt)
SELECT id, date, name, status, amt
FROM orders__dbt_tmp;

      

    

    
Edge cases

    
    
      
  • Source has a late-arriving row (e.g. order placed on Jan 8 but arrived Jan 12): if your filter is WHERE date > MAX(date) the late row is silently dropped forever because its date is older than the cursor. Use updated_at instead of event date when possible.
    • 
      
  • Restatement needed: append can't correct old rows. If order 2's status changes from pending → paid, the stale row stays. You must do a full refresh.
    • 
      
  • Filter uses same column as partition on BigQuery: WHERE date > MAX(date) requires a full scan of the target to compute MAX — negating partition pruning. Use a fixed lookback instead: WHERE date >= DATE_SUB(CURRENT_DATE, INTERVAL 3 DAY).
    • 
      
  • No filter + append = geometric duplication: run N times → N copies of every row. Each full-refresh doubles the existing rows. Common in pipelines where is_incremental() block was accidentally deleted.
    • 
    

  


  
  

    
merge

    
Macro: get_merge_sql → MERGE INTO

    

      

        
dbt model


{{ config(
  materialized='incremental',
  unique_key='id',
  incremental_strategy='merge'
) }}

SELECT id, date, name, status, amt
FROM orders_source
{% if is_incremental() %}
  WHERE updated_at > (
    SELECT MAX(updated_at) FROM {{ this }})
{% endif %}

      

      

        
compiled SQL (incremental run)


-- Step 1: temp table (rows 2, 5, 6)
CREATE TABLE orders__dbt_tmp AS
SELECT * FROM orders_source
WHERE updated_at > (SELECT MAX(updated_at)
                    FROM orders);

-- Step 2: merge (get_merge_sql)
MERGE INTO orders AS DBT_INTERNAL_DEST
USING orders__dbt_tmp AS DBT_INTERNAL_SOURCE
  ON DBT_INTERNAL_DEST.id = DBT_INTERNAL_SOURCE.id
WHEN MATCHED THEN UPDATE SET
  name   = DBT_INTERNAL_SOURCE.name,
  status = DBT_INTERNAL_SOURCE.status,
  amt    = DBT_INTERNAL_SOURCE.amt,
  date   = DBT_INTERNAL_SOURCE.date
WHEN NOT MATCHED THEN INSERT
  (id, date, name, status, amt)
  VALUES
  (DBT_INTERNAL_SOURCE.id, ...);

      

    

    
With merge_update_columns: ['status','amt']


WHEN MATCHED THEN UPDATE SET
  status = DBT_INTERNAL_SOURCE.status,   -- only these two columns
  amt    = DBT_INTERNAL_SOURCE.amt
  -- name and date are NOT updated even if source changed them

    
With incremental_predicates


MERGE INTO orders AS DBT_INTERNAL_DEST
USING orders__dbt_tmp AS DBT_INTERNAL_SOURCE
  ON  DBT_INTERNAL_DEST.id = DBT_INTERNAL_SOURCE.id
  AND DBT_INTERNAL_DEST.date > dateadd(day,-7,current_date())
  -- rows OUTSIDE this window are never touched (not even scanned)

    
Edge cases

    
    
      
  • unique_key on multiple columns: pass a list unique_key=['order_id','line_item']. dbt generates ON DEST.order_id = SRC.order_id AND DEST.line_item = SRC.line_item. If either column has NULLs, the ON condition is never true (SQL NULL ≠ NULL) so duplicates are inserted silently.
    • 
      
  • No unique_key: dbt omits the WHEN MATCHED clause entirely. The MERGE only inserts — identical to append. A common misconfiguration.
    • 
      
  • merge_update_columns AND merge_exclude_columns together: dbt raises a compilation error — they are mutually exclusive.
    • 
      
  • Predicate misses a changed row: if order 0 (Zara, 2024-01-01) is updated in source but the predicate window is date > 2024-01-08, the merge never sees row 0. The old value stays in the target forever with no error.
    • 
      
  • Full table scan without predicates: on a 100M row table, the MERGE ON clause scans all rows to find matches. Predicates reduce this to only the recent partition — often a 100× speedup.
    • 
      
  • Snowflake: merge inserts duplicate source rows: if the temp table has two rows with the same unique_key (e.g. event dedup failed upstream), Snowflake's MERGE errors with "non-deterministic MERGE". BigQuery silently picks one. Add deduplication in your model SQL.
    • 
    

  


  
  

    
delete+insert

    
Macro: get_delete_insert_merge_sql → DELETE + INSERT (two statements)

    

      

        
dbt model


{{ config(
  materialized='incremental',
  unique_key='id',
  incremental_strategy='delete+insert'
) }}

SELECT id, date, name, status, amt
FROM orders_source
{% if is_incremental() %}
  WHERE updated_at > (
    SELECT MAX(updated_at) FROM {{ this }})
{% endif %}

      

      

        
compiled SQL (two separate statements)


-- Step 1: temp table
CREATE TABLE orders__dbt_tmp AS
SELECT * FROM orders_source
WHERE updated_at > (SELECT MAX(updated_at)
                    FROM orders);

-- Step 2a: DELETE matching rows
DELETE FROM orders
WHERE id IN (SELECT id FROM orders__dbt_tmp);

-- Step 2b: INSERT all temp rows
INSERT INTO orders (id, date, name, status, amt)
SELECT id, date, name, status, amt
FROM orders__dbt_tmp;

      

    

    
With incremental_predicates — DELETE is scoped


-- Step 2a becomes:
DELETE FROM orders
WHERE id IN (SELECT id FROM orders__dbt_tmp)
  AND date > dateadd(day,-7,current_date())
-- Step 2b still inserts ALL temp rows regardless of predicate
-- → rows outside predicate window are re-inserted but never deleted → DUPLICATES

    
Edge cases

    
    
      
  • Not atomic: DELETE and INSERT are two separate statements. A job failure between them leaves matched rows deleted with nothing in their place — data loss until re-run. Merge is atomic; delete+insert is not.
    • 
      
  • Predicate + no is_incremental filter = duplicates: without the filter, old rows appear in the temp table. The predicate blocks the DELETE for rows outside the window, but the INSERT still re-inserts them — creating two copies. Visible in the simulator above.
    • 
      
  • No unique_key: dbt errors at compile time — delete+insert needs a unique_key to know what to DELETE. Unlike merge (which silently falls back to append), this is a hard error.
    • 
      
  • unique_key is a list: generates WHERE (col1, col2) IN (SELECT col1, col2 FROM tmp). Row-value constructors are not supported on all platforms — verify on your adapter first.
    • 
      
  • Slower than merge for row-level updates: every matched row is deleted then re-inserted (2 write ops). Merge updates in-place (1 write op). Delete+insert shines when the adapter doesn't support MERGE (e.g. some Redshift configurations).
    • 
    

  


  
  

    
insert_overwrite

    
Macro: get_insert_overwrite_merge_sql → INSERT OVERWRITE (partition-level replace)

    

      

        
dbt model (BigQuery)


{{ config(
  materialized='incremental',
  incremental_strategy='insert_overwrite',
  partition_by={
    'field': 'date',
    'data_type': 'date',
    'granularity': 'day'
  }
) }}

SELECT id, date, name, status, amt
FROM orders_source
{% if is_incremental() %}
  WHERE date >= DATE_SUB(CURRENT_DATE, INTERVAL 3 DAY)
{% endif %}

      

      

        
compiled SQL — dynamic partition mode


-- Step 1: temp table (dates: 2024-01-09, 01-10, 01-11)
CREATE TABLE orders__dbt_tmp
PARTITION BY DATE(date) AS
SELECT * FROM orders_source
WHERE date >= DATE_SUB(CURRENT_DATE, INTERVAL 3 DAY);

-- Step 2: overwrite only the partitions present in temp
INSERT INTO orders
  OVERWRITE PARTITION (date='2024-01-09'),
            PARTITION (date='2024-01-10'),
            PARTITION (date='2024-01-11')
SELECT * FROM orders__dbt_tmp;

      

    

    
Static partition mode — you declare partitions explicitly


{{ config(
  partitions=["'2024-01-10'", "'2024-01-11'"]   -- hardcoded, must match data type
) }}

-- compiled: overwrites exactly those two dates even if temp has other dates
INSERT INTO orders OVERWRITE PARTITION (date='2024-01-10'), PARTITION (date='2024-01-11')
SELECT * FROM orders__dbt_tmp;

    
Edge cases

    
    
      
  • Source has a new row on an OLD partition date: if a late-arriving 2024-01-01 row arrives today and the filter only selects last 3 days, the row never enters the temp table → never overwrites the old partition → silently dropped. Use a longer lookback for event-time data with delays.
    • 
      
  • Dynamic mode is destructive at partition level: if the temp table selects 2024-01-10 rows, the entire 2024-01-10 partition is replaced — including rows that weren't in the source query. This is intentional but surprising when you share a partition across multiple models.
    • 
      
  • Static mode doesn't validate: if you declare partitions=["'2024-01-10'"] but your model selects 2024-01-11 rows too, they are overwritten (on some platforms) or silently dropped depending on partition mode. Always match your SELECT filter to your partitions list.
    • 
      
  • BigQuery only supports this natively: on Spark/Databricks, insert_overwrite uses INSERT OVERWRITE with dynamic partition mode. On Snowflake it's not supported — use merge or delete+insert instead.
    • 
      
  • No unique_key: insert_overwrite does not use unique_key at all. Within a partition, all rows from the source replace all rows in the target — no row-level dedup. If you write two rows with the same id into one partition, both will be there.
    • 
    

  


  
  

    
microbatch

    
Macro: get_incremental_microbatch_sql — each adapter provides its own; default raises not_implemented_for_adapter

    

      

        
dbt model


{{ config(
  materialized='incremental',
  incremental_strategy='microbatch',
  event_time='created_at',
  begin='2024-01-01',
  batch_size='day',
  lookback=1
) }}

SELECT id, created_at, name, status, amt
FROM orders_source
-- NO is_incremental() filter needed —
-- dbt injects the batch window automatically

      

      

        
compiled SQL per batch (e.g. 2024-01-10)


-- dbt generates one run per day between begin and now
-- For batch date = 2024-01-10:

-- Step 1: temp table — batch-scoped source
CREATE TABLE orders__dbt_tmp AS
SELECT * FROM orders_source
WHERE created_at >= '2024-01-10'
  AND created_at <  '2024-01-11';

-- Step 2: overwrite that batch window in target
DELETE FROM orders
WHERE created_at >= '2024-01-10'
  AND created_at <  '2024-01-11';

INSERT INTO orders
SELECT * FROM orders__dbt_tmp;

      

    

    
lookback=1 in practice


-- On 2024-01-12, with lookback=1 dbt reprocesses yesterday too:
-- Batch 1: 2024-01-11 (yesterday — catches late-arriving rows)
-- Batch 2: 2024-01-12 (today)
-- Without lookback (lookback=0): only processes today

    
Edge cases

    
    
      
  • event_time must exist in the model SELECT: dbt uses it for both the temp table filter and the target DELETE. If you alias it (created_at AS event_ts), configure event_time='event_ts' — not the source column name.
    • 
      
  • begin is inclusive and permanent: on the first full run, dbt backfills from begin to now. With 3 years of data and batch_size='day' that's ~1095 individual SQL round-trips. On expensive warehouses this adds up quickly — set begin as recent as your use case allows.
    • 
      
  • Batch failures are independent: unlike append/merge (one run = one statement), a microbatch failure on 2024-01-10 doesn't affect 2024-01-11. dbt tracks batch status — failed batches are retried on the next run only if you enable --retry-failed or re-run manually.
    • 
      
  • is_incremental() always returns true inside a batch: there's no "full refresh" mode in the traditional sense — you control the window via begin and the run date. To re-process all history, change begin to an earlier date.
    • 
      
  • Not available on all adapters: BigQuery, Snowflake, Databricks, Postgres support it. Check your adapter version — microbatch was introduced in dbt 1.9.
    • 
    

  







  

    
Incremental Predicates

    
Limiting the scan of the target table

    
Without predicates, every merge scans the entire target table. With predicates, you restrict which rows dbt even looks at.

  


  

    
The problem: full table scans on merge

    
When dbt runs a merge, it generates SQL like:


-- Without predicates: scans ALL 500M rows in the target
MERGE INTO orders AS DBT_INTERNAL_DEST
USING orders__dbt_tmp AS DBT_INTERNAL_SOURCE
  ON DBT_INTERNAL_DEST.order_id = DBT_INTERNAL_SOURCE.order_id
WHEN MATCHED THEN UPDATE ...
WHEN NOT MATCHED THEN INSERT ...

    
The ON clause forces the engine to join every row in the target against every row in the temp table. For a 500M-row table, that's a full scan on every incremental run.

  


  

    
The fix: incremental_predicates

    
Add a predicate that limits which rows of the target are even considered:


-- With incremental_predicates: only scans last 7 days in target
MERGE INTO orders AS DBT_INTERNAL_DEST
USING orders__dbt_tmp AS DBT_INTERNAL_SOURCE
  ON  DBT_INTERNAL_DEST.order_id = DBT_INTERNAL_SOURCE.order_id
  AND DBT_INTERNAL_DEST.order_date > dateadd(day, -7, current_date())   ← predicate
WHEN MATCHED THEN UPDATE ...
WHEN NOT MATCHED THEN INSERT ...

    
Important: The predicate constrains which rows in the target participate in the merge. Rows outside the predicate window are untouched — dbt can't update them even if the source changed. Only use this when you're confident late-arriving changes won't fall outside the window.

The two aliases inside a predicate

    

      
AliasWhat it refers toUse it to…

Predicates vs. is_incremental() — two different filters

    
These address different sides of the merge. Don't confuse them.

    

      

        
is_incremental() filter

        
Limits what goes into the source / temp table. Controls how much new data dbt even fetches.


{% if is_incremental() %}
  where updated_at >
    (select max(updated_at)
     from {{ this }})
{% endif %}

      

      

        
incremental_predicates

        
Limits which rows in the target table are scanned during the merge join.


incremental_predicates=[
  "DBT_INTERNAL_DEST.session_start
    > dateadd(day,-7,current_date)"
]

      

    

    
Best practice: Use both together on large tables — is_incremental() shrinks the source temp table, predicates shrink the target scan. This minimises data movement on both sides of the merge.

How dbt resolves a strategy

    
When dbt runs an incremental model it calls adapter.get_incremental_strategy_macro() to look up the macro named get_incremental_<strategy>_sql. That macro receives a single arg_dict dict and returns a SQL string.


-- incremental.sql (simplified from dbt-core)
{% set incremental_strategy = config.get('incremental_strategy') or 'default' %}
{% set incremental_predicates = config.get('predicates', none)
                              or config.get('incremental_predicates', none) %}

{% set strategy_arg_dict = {
    'target_relation':      target_relation,
    'temp_relation':        temp_relation,
    'unique_key':           unique_key,
    'dest_columns':         dest_columns,
    'incremental_predicates': incremental_predicates
} %}

{% set strategy_sql_macro_func =
    adapter.get_incremental_strategy_macro(context, incremental_strategy) %}
{% set build_sql = strategy_sql_macro_func(strategy_arg_dict) %}

  


  
  

    
What most people miss

    
Five things about dbt internals that cause real production bugs — none of them obvious from the docs.


    


      

        

          1
          on_schema_change is completely separate from your strategy
        

        
Your incremental strategy controls how rows are merged. It knows nothing about columns. If you add a new column to your model, what happens to the target table depends entirely on on_schema_change — a separate config the strategy never touches.

        

          
ignore (default)
New columns in model are silently dropped from INSERT. Your data is missing a column and you get no error.

          
fail
Run errors immediately if model and target schemas differ. Safe but requires manual intervention.

          
append_new_columns
Adds new columns to the target. Existing rows get NULL for the new column. Removed columns stay.

          
sync_all_columns
Adds new and drops removed columns. The most aggressive option — verify before using in production.

        

        
The silent killer: The default is ignore. You add a revenue column to your model, run dbt, no error — but every row in the target has NULL for revenue because the column was never added to the target table. You only notice on the dashboard.

      


      

        

          2
          The temp table is a real warehouse table — and it can be left behind
        

        
dbt creates orders__dbt_tmp as an actual table in your warehouse before running the strategy SQL. If the run fails mid-way, this table is left behind. On the next run, dbt tries to create it again and gets a "relation already exists" error — which then masks the real error from the failed run.


-- What dbt actually runs before your strategy SQL:
CREATE TABLE orders__dbt_tmp AS
SELECT ... FROM your_model_sql;

-- If this step succeeds but the MERGE fails, __dbt_tmp persists.
-- Next run: "RuntimeError: Relation already exists: orders__dbt_tmp"

        
Fix: dbt Cloud retries clean up temp tables automatically. On dbt Core, manually DROP TABLE IF EXISTS orders__dbt_tmp in your warehouse before retrying, or use dbt run --full-refresh which rebuilds from scratch.

      


      

        

          3
          is_incremental() returns false on the first run — and on every full refresh
        

        
The macro checks whether the target relation already exists. If this is the first time you're running the model, or if you've run dbt run --full-refresh (which drops the table), is_incremental() returns false and your WHERE filter is skipped entirely — the model runs as a full load.


-- is_incremental.sql (actual source):
{% macro is_incremental() %}
  {%- if this in graph.nodes and
      config.get('materialized') == 'incremental' and
      not flags.FULL_REFRESH -%}
    {{ return(True) }}
  {%- else -%}
    {{ return(False) }}
  {%- endif -%}
{% endmacro %}

        
Implication: --full-refresh is not just a "reset" — it drops and rebuilds the whole table from scratch, scanning your entire source. On a 5-year event table this can be expensive. Use it deliberately, not as a default troubleshooting step.

      


      

        

          4
          Adapters silently override built-in strategies
        

        
When you specify incremental_strategy='merge' on BigQuery, you are not running the global get_incremental_merge_sql macro. BigQuery's adapter ships its own override that adds partition-aware merge logic. The dispatch chain is: your project → installed packages → adapter → global project. The first match wins.


-- Resolution order for get_incremental_merge_sql:
-- 1. macros/ in your project   (your custom override, if any)
-- 2. installed packages        (e.g. dbt-utils override)
-- 3. adapter macro namespace   (e.g. dbt-bigquery's version)
-- 4. global project            (the built-in fallback)

-- This is why merge behaves differently on BigQuery vs Snowflake
-- even with identical model config.

        
Debugging tip: Run dbt compile and inspect the compiled SQL in target/compiled/. That's the actual SQL that will execute — it removes all ambiguity about which macro won the dispatch race.

      


      

        

          5
          unique_key is not a database constraint — duplicates in source cause silent data corruption
        

        
dbt uses unique_key purely to generate the ON clause of a MERGE statement. It does not enforce uniqueness in the source or the target. If your temp table contains two rows with the same unique_key value, each platform handles this differently — and only one of them tells you about it.

        

          
Snowflake
Raises "non-deterministic MERGE" error. Safe — you know immediately.

          
BigQuery
Silently picks one of the duplicate rows. Which one? Undefined. Your data is wrong and you get no error.

          
Spark / Databricks
Default behaviour varies by config. May apply both updates sequentially, leaving the last one — or raise an error depending on spark.sql.optimizer.canChangeMergeIntoConfig.

        

        
Fix: Deduplicate in your model SQL before the strategy runs: SELECT * FROM source QUALIFY ROW_NUMBER() OVER (PARTITION BY id ORDER BY updated_at DESC) = 1. Never rely on the warehouse to handle source duplicates gracefully.

When do you actually need a custom strategy?

    
The five built-in strategies cover the majority of cases. You reach for a custom strategy when a built-in assumption breaks down for your specific data, platform, or compliance requirement.

    

      

        
Your unique_key has NULLs

        
SQL NULL ≠ NULL, so a standard MERGE ON key = key never matches NULL rows. A custom strategy can use IS NOT DISTINCT FROM (null-safe equality) in the ON clause — something the built-in merge macro doesn't expose.

      

      

        
You need a soft delete

        
Built-in strategies physically delete or overwrite rows. If your compliance requirement is to keep deleted records with a deleted_at timestamp and is_deleted=true flag, no built-in strategy does this — you need a custom MERGE that sets those columns instead of deleting rows.

      

      

        
You need to preserve specific columns on update

        
merge_update_columns and merge_exclude_columns are all-or-nothing whitelists/blacklists. A custom strategy can write conditional UPDATE logic: e.g. only overwrite status if the new value is a valid forward transition (pending → paid, never paid → pending).

      

      

        
Platform quirks

        
Some platforms require non-standard syntax: Redshift's MERGE variant, Teradata's UPSERT, or Oracle's MERGE ... MATCHED UPDATE WHERE conditional. A custom strategy generates the exact dialect your warehouse expects.

      

      

        
You want to run multiple statements

        
Built-in strategies produce a single SQL statement. A custom strategy can return a list of statements: e.g. DELETE expired rows, then MERGE new ones, then UPDATE a watermark table — all as one atomic operation under the same dbt run.

      

      

        
Shared pattern across many models

        
If 20 models all need the same bespoke upsert logic, copy-pasting SQL into every model is error-prone. A custom strategy encapsulates it once in a macro and lets every model reference it by name in config — a single place to maintain.

      

    

  


  
  

    

      Example 1
      
Null-safe merge

    

    
Use case: Customer records where device_id can be NULL — a retail loyalty programme syncs customer profiles from a mobile app. Some users registered without a device (device_id is NULL). Standard MERGE skips them silently because NULL = NULL is false.

    

      

        
macros/incremental_null_safe.sql


{% macro get_incremental_null_safe_sql(arg_dict) %}
  {%- set target = arg_dict["target_relation"] -%}
  {%- set source = arg_dict["temp_relation"] -%}
  {%- set uk     = arg_dict["unique_key"] -%}

  merge into {{ target }} as DEST
  using {{ source }} as SRC
    on  DEST.{{ uk }}
        is not distinct from
        SRC.{{ uk }}     -- NULL-safe equality
  when matched then update set
    email      = SRC.email,
    tier       = SRC.tier,
    updated_at = SRC.updated_at
  when not matched then insert
    ({{ uk }}, email, tier, updated_at)
    values
    (SRC.{{ uk }}, SRC.email, SRC.tier, SRC.updated_at)
{% endmacro %}

      

      

        
models/customers.sql


{{ config(
  materialized='incremental',
  incremental_strategy='null_safe',
  unique_key='device_id'
) }}

select
    device_id,   -- can be NULL
    email,
    tier,
    updated_at
from {{ ref('raw_customers') }}
{% if is_incremental() %}
    where
        updated_at
        > (select max(updated_at) from {{ this }})
{% endif %}

        
Why not use merge? Standard merge ON device_id = device_id evaluates to NULL (unknown) when either side is NULL — those rows are never matched and always inserted as duplicates. IS NOT DISTINCT FROM treats NULL = NULL as true.

      

    

  


  
  

    

      Example 2
      
Soft delete (GDPR-safe audit trail)

    

    
Use case: Order management system with compliance requirements — a marketplace must retain a record of every order for 7 years but mark cancelled orders as deleted for the product team's reporting. Hard deletes violate the audit trail.

    

      

        
macros/incremental_soft_delete.sql


{% macro get_incremental_soft_delete_sql(arg_dict) %}
  {%- set target = arg_dict["target_relation"] -%}
  {%- set source = arg_dict["temp_relation"] -%}

  merge into {{ target }} as DEST
  using {{ source }} as SRC
    on DEST.order_id = SRC.order_id
  when matched
    and SRC.status = 'cancelled'
    then update set
      DEST.is_deleted  = true,
      DEST.deleted_at  = SRC.updated_at,
      DEST.status      = SRC.status
  when matched
    and SRC.status != 'cancelled'
    then update set
      DEST.status      = SRC.status,
      DEST.updated_at  = SRC.updated_at
  when not matched then insert
    (order_id, status, is_deleted,
     deleted_at, created_at, updated_at)
    values
    (SRC.order_id, SRC.status, false,
     null, SRC.created_at, SRC.updated_at)
{% endmacro %}

      

      

        
models/orders.sql


{{ config(
  materialized='incremental',
  incremental_strategy='soft_delete',
  unique_key='order_id'
) }}

select
    order_id,
    status,
    created_at,
    updated_at
from {{ ref('raw_orders') }}
{% if is_incremental() %}
    where
        updated_at
        > (select max(updated_at) from {{ this }})
{% endif %}

        
Why not delete+insert? delete+insert physically removes the row then re-inserts it. That loses the original created_at and any audit columns not in the source. The soft delete strategy keeps all history intact and only flips the is_deleted flag.

      

    

  


  
  

    

      Example 3
      
SCD Type 2 — full history tracking

    

    
Use case: Subscription pricing history for a SaaS platform — finance needs to know what price a customer was on at any point in time for revenue recognition. Standard merge overwrites the old price, losing history.

    

      

        
macros/incremental_scd2.sql


{% macro get_incremental_scd2_sql(arg_dict) %}
  {%- set target = arg_dict["target_relation"] -%}
  {%- set source = arg_dict["temp_relation"] -%}

  -- Step 1: close the current row for changed records
  update {{ target }} as DEST
  set
    valid_to    = SRC.valid_from,
    is_current  = false
  from {{ source }} as SRC
  where DEST.account_id = SRC.account_id
    and DEST.is_current  = true
    and DEST.monthly_price != SRC.monthly_price;

  -- Step 2: insert new current row (changed + brand new)
  insert into {{ target }}
    (account_id, monthly_price,
     valid_from, valid_to, is_current)
  select
    SRC.account_id,
    SRC.monthly_price,
    SRC.valid_from,
    null,         -- open-ended: no end date yet
    true
  from {{ source }} as SRC
  left join {{ target }} as DEST
    on  DEST.account_id = SRC.account_id
    and DEST.is_current = true
  where DEST.account_id is null          -- net new accounts
     or DEST.monthly_price
        != SRC.monthly_price;            -- price changed
{% endmacro %}

      

      

        
models/subscription_history.sql


{{ config(
  materialized='incremental',
  incremental_strategy='scd2',
  unique_key='account_id'
) }}

select
    account_id,
    monthly_price,
    current_timestamp() as valid_from
from {{ ref('raw_subscriptions') }}
{% if is_incremental() %}
    where
        updated_at
        > (select max(valid_from) from {{ this }})
{% endif %}

        

          
Target table after two price changes:

          

            
account_idpricevalid_fromvalid_tocurrent

            
acct_1$492024-01-012024-06-01false

            
acct_1$792024-06-012024-11-01false

            
acct_1$992024-11-01nulltrue ✓

          

        

        
Why not merge? Merge overwrites the single row — you'd only ever see the latest price. SCD2 keeps every historical row, closing the old one and opening a new one. Finance can query WHERE valid_from <= report_date AND (valid_to > report_date OR valid_to IS NULL) to get the price at any point in time.

      

    

  


  
  

    
How dbt finds your custom strategy

    
No registration needed. Name your macro get_incremental_<strategy_name>_sql and dbt resolves it by name at compile time via adapter.get_incremental_strategy_macro().


-- macros/my_strategies.sql  (anywhere in your macros/ folder)
{% macro get_incremental_my_strategy_sql(arg_dict) %}
  -- arg_dict always contains:
  --   target_relation      → the existing table
  --   temp_relation        → dbt's temp table this run
  --   unique_key           → from model config
  --   dest_columns         → list of column objects
  --   incremental_predicates → list of predicate strings
  {% do return(my_strategy_sql(
      arg_dict["target_relation"],
      arg_dict["temp_relation"]
  )) %}
{% endmacro %}

-- Then in any model:
{{ config(
    materialized='incremental',
    incremental_strategy='my_strategy'
) }}

    
Delegating to a package: If a community package ships the strategy, re-export it with a one-liner: {% macro get_incremental_null_safe_sql(arg_dict) %}{% do return(mypackage.get_incremental_null_safe_sql(arg_dict)) %}{% endmacro %}
